# Роадмап: что сделано и куда двигаться дальше

Живой документ: сделали фичу — переносим её из планов наверх, чтобы
было видно, как проект растёт. Если чего-то не хватает вам лично —
это хороший кандидат на следующий пункт.

## ✅ Реализовано (v0.9)

Настройки «налету», подробные логи и умный троттлинг:

- **Адаптивный троттлинг по 429** — общий cooldown-gate на все потоки:
  один rate-limit ставит на паузу сразу всех воркеров (уважается
  Retry-After, штраф растёт экспоненциально и спадает после успеха).
  Раньше N параллельных потоков продолжали жечь квоту по одному.
- **xelatex-движок PDF** — если установлен TeX (xelatex/tectonic/
  lualatex), наш LaTeX-экспорт компилируется в PDF: настоящая
  типографика без pandoc и браузера. Порядок движков: pandoc →
  chromium → xelatex → weasyprint.
- **Режим подробных логов** — `PDFTRANSL_LOG_LEVEL=DEBUG` включает
  полную телеметрию везде (CLI, бот, Django-сервер): каждый LLM-вызов
  с размерами/временем/токенами, ход сегментов, решения парсера и
  экспорта. `PDFTRANSL_LOG_FILE` дублирует в файл с ротацией. Уровень
  меняется и на лету из веб-настроек.
- **Веб: вкладки и настройки «налету»** — панель серверных настроек
  (провайдер, модели, парсер, воркеры, rpm, форматы, поведение
  пайплайна, уровень логов) сохраняется в БД и действует на все новые
  задачи без перезапуска; per-job опции приоритетнее. Плюс вкладка
  глоссария (просмотр/добавление/удаление терминов) и удаление задач
  вместе с файлами результата.

## ✅ Реализовано (v0.8)

Устойчивость парсинга на крупных файлах:

- **Цепочка фолбэков парсеров** — если основной парсер падает (типично:
  MinerU уходит в свой таймаут на большом скане, на Mac без CUDA он
  считает OCR на CPU и не успевает), задача больше не рушится с
  трейсбеком. Пайплайн автоматически пробует следующий доступный бэкенд;
  при наличии мультимодальной модели откатывается на VLM-OCR (лучше
  PyMuPDF на сканах). Использованный бэкенд и причина отката видны в
  отчёте и UI.
- **Таймаут локального парсера** (`PDFTRANSL_PARSER_TIMEOUT`, 30 мин по
  умолчанию) — зависший MinerU принудительно останавливается, а не висит
  вечно; в сообщении понятная причина и подсказки.
- Отключается через `PDFTRANSL_PARSER_FALLBACK=false` (тогда падение
  парсера даёт чистый статус «failed» с ошибкой, без сырого краша).

## ✅ Реализовано (v0.7)

Мультимодальная основная модель = OCR из коробки:

- **Авто-определение vision у основной модели** — раньше OCR включался
  только при отдельно заданной `vision_model`, поэтому у пользователя с
  мультимодальной основной моделью (gemma3:12b) битый PDF всё равно шёл
  в текстовую обработку → точки и транслит. Теперь по имени модели
  (gemma3, llava, `*-vl`, llama3.2-vision, minicpm-v, gpt-4o, claude…)
  определяется мультимодальность, и OCR использует ту же основную
  модель, не требуя отдельной настройки.
- **OCR-промпт запрещает транслитерацию** — «сохраняй исходный язык и
  письменность, кириллицу оставляй кириллицей» (лечит «странный
  транскриб русского на английском»).
- **Память переводов не учится на битом/сканированном источнике** без
  OCR — иначе мусор попадал в TM и повторная попытка выдавала те же
  кракозябры из точного совпадения.
- UI подсказывает, что мультимодальная модель сама распознаёт сканы.

## ✅ Реализовано (v0.6)

Диагностика «кракозябр» и неверного направления:

- **Детектор битого текстового слоя** (`parsing/text_quality.py`) —
  ловит PDF, где шрифты без ToUnicode-карты дают мусор (символы Private
  Use Area, «·····», replacement, latin1-mojibake). Типичная беда
  Cyberleninka-подобных русских PDF: страница выглядит нормально, а
  извлекается хлам. Детектор сканов это пропускал (текст формально
  есть) — теперь `needs_ocr = скан ИЛИ битый текст`, и такие документы
  авто-роутятся в OCR (или получают предупреждение + статус «partial»
  вместо ложного «completed» с кракозябрами на выходе).
- **Проверка направления перевода** — если текст в основном кириллица,
  а язык оригинала стоит `en` (частая ошибка «забыл переключить
  RU↔EN»), в отчёте и UI появляется предупреждение. Устойчиво к
  вкраплениям англ-терминов в русский текст.

## ✅ Реализовано (v0.5)

Довели vision и починили экспорт-движки:

- **Честная детекция PDF-движка** — `available_engines()` теперь
  проверяет, что браузер Chromium реально скачан (а не только что
  установлен пакет playwright). Раньше UI показывал PDF доступным, а
  файл молча не собирался. В форме — метка «нет движка», в панели
  задачи — баннер с причиной, если формат запросили, а движка нет.
- **quickstart ставит Chromium** (`playwright install chromium`) и
  Dockerfile тоже — PDF с формулами работает из коробки.
- **Даунскейл изображений для VLM** — страницы сканов и рисунки
  ужимаются до 2200px (Pillow) перед отправкой в модель: мегабайты →
  десятки КБ, меньше шанс упереться в лимиты провайдера. Общий хелпер
  `vision_message()`; VLM-OCR получил ретрай на страницу.
- **Форматы по умолчанию** — `html, docx, pdf` (было только `html`),
  так что PDF собирается без явного `--formats`.
- **latex в rebuild**, описания рисунков и OCR-инфо — в QA-отчёте;
  vision-клиент кэшируется и переиспользуется для OCR и рисунков.

## ✅ Реализовано (v0.4)

Читабельные формулы во всех форматах и поддержка сканов:

- **Офлайн-формулы в HTML/PDF** — KaTeX вшивается в файл (инлайн CSS+JS,
  шрифты data-URI); HTML полностью автономен, Chromium-PDF рендерит
  формулы без интернета. Раньше офлайн формулы оставались сырым `$...$`.
- **Читабельные формулы в DOCX без pandoc** — рендер в картинки через
  matplotlib mathtext (вне подмножества — фолбэк в текст); плюс защита
  от control-символов, ронявших запись DOCX.
- **Распознавание сканов** — детектор image-only PDF
  (`parsing/scan_detect.py`) и VLM-OCR бэкенд (`--backend vlm_ocr` или
  авто-роутинг): страницы → картинки → vision-модель → Markdown+LaTeX.
  Работает локально (qwen2.5-vl) и в облаке. Нет vision-модели —
  предупреждение в отчёте/UI вместо молча пустого перевода.
- **latex в rebuild** — пересборка после правок больше не теряет .tex.
- Docker: KaTeX dist копируется в runtime-образ (`PDFTRANSL_KATEX_DIR`).

## ✅ Реализовано (v0.3)

Вторая волна улучшений:

- **Новые парсеры**: marker и Docling как бэкенды (`--backend marker`),
  auto-выбор: MinerU → marker → Docling → PyMuPDF.
- **LLM-починка битых формул** — сломанный OCR-ом LaTeX чинится моделью
  и принимается только если проходит ту же синтаксическую проверку.
- **LLM-судья качества** (`--score`) — оценка 0–100 на сегмент, слабые
  уходят на ревью; сводка в отчёте.
- **Structured outputs** (`--structured`) — JSON-mode для ревью,
  скоринга и извлечения терминов на OpenAI-совместимых серверах.
- **Rate-limiter** (`--rpm N`) — общий бюджет запросов/минуту на всю
  цепочку провайдеров; спасает бесплатные тарифы.
- **Рендер-проверка KaTeX** (`--render-check`) — итоговый HTML
  открывается в headless Chromium, ошибки рендера формул — в отчёт.
- **Экспорт в LaTeX** (`--formats latex`) — компилируемый .tex проект.
- **numpy-ускорение** поиска по памяти переводов.
- **Глоссарий учится на правках** — короткая правка-термин из вычитки
  автоматически становится записью глоссария.
- **SSE-прогресс** (`/api/jobs/<id>/events/`) — фронтенд слушает стрим
  вместо поллинга (поллинг остался как фолбэк).
- **Токен-аутентификация API** (`PDFTRANSL_API_TOKEN`) и per-IP
  троттлинг загрузок.
- **quickstart.sh** — окружение, зависимости, фронтенд, настройка под
  Ollama одной командой; гайды docs/OLLAMA.md и docs/FINETUNING.md.

## ✅ Реализовано (v0.2)

Из роадмапа v0.1 в движок вошли:

- **Параллельный перевод сегментов** — ThreadPoolExecutor,
  `max_workers` (кратное ускорение на длинных статьях); контекст с
  исходной стороны, поэтому параллельность не ломает связность.
- **Fallback-цепочка провайдеров** — `FallbackClient`
  (`--fallback openrouter` / `PDFTRANSL_FALLBACK_PROVIDERS`): локальная
  модель с облачной подстраховкой и наоборот.
- **Авто-глоссарий документа** — LLM извлекает термины из начала статьи,
  переводы принудительно подставляются во все сегменты.
- **Саммари документа** в system-промпте — модель знает, о чём статья.
- **Контекст между сегментами** — хвост предыдущего исходного сегмента.
- **Детекция References/Bibliography/Списка литературы** — библиография
  не переводится (сохраняется цитируемость), Appendix после неё — снова
  переводится.
- **Back-translation чек** (опция) — обратный перевод + косинус
  эмбеддингов, предупреждение о потере смысла.
- **LaTeX-проверка** итогового документа — балансы скобок/окружений,
  битые формулы в отчёте.
- **Кэш парсинга** по SHA-256 содержимого PDF.
- **Доменные теги** в translation memory (фильтр поиска).
- **Двуязычный режим** — документ «оригинал-цитата → перевод».
- **Экспорт PDF/DOCX/HTML** с репликой вёрстки: pandoc (нативные
  формулы Word) → python-docx; pandoc+xelatex → Chromium → weasyprint;
  автономный HTML с KaTeX и data-URI картинками.
- **Человек в цикле** — веб-вычитка по сегментам (side-by-side), правки
  уходят в TM с приоритетом, пересборка всех форматов одной кнопкой.
- **Django-бэкенд** — jobs API, прогресс по стадиям, Celery или
  фоновые потоки, админка; **React SPA**; **Telegram-бот** (aiogram);
  **docker-compose** (web + worker + redis + bot + ollama).
- **Кэш/лимиты загрузки** в API (размер, тип файла).

## ✅ Реализовано (v0.12)

Пауза задач и устойчивость перевода к затупам LLM:

- **Пауза/возобновление задач** — кнопка «Пауза» в веб-интерфейсе
  (`POST /api/jobs/<id>/pause/`) останавливает перевод кооперативно:
  между сегментами (последовательный режим) или сразу после уже
  запущенных, отменяя ещё не начатые (параллельный режим,
  `ThreadPoolExecutor.cancel()`). Уже переведённая часть документа сразу
  доступна для скачивания. «Продолжить перевод»
  (`POST /api/jobs/<id>/resume/`) не начинает заново — используется тот
  же чекпойнт-механизм, что и восстановление после краша, так что
  переводятся только оставшиеся сегменты.
- **Исправлен баг «пустой документ после успешного перевода»** — если
  LLM отвечала на сегмент пустой строкой (зависание, перегрузка модели),
  это раньше протекало в собранный документ как пустой абзац:
  `Segment.final_text()` проверял перевод на `is not None`, а не на
  истинность, так что пустая-но-не-`None` строка не откатывалась на
  исходный текст. Теперь откатывается — независимо от паузы.
- **Дискретная запись + устойчивость к сбоям обогащения** — переведённый
  документ пишется на диск сразу после стадии перевода, а не один раз в
  самом конце пайплайна. Каждая из последующих необязательных стадий
  (scoring, review, back-translation, LaTeX-починка, копирование
  ассетов, описание рисунков, экспорт, render-check, TM) обёрнута в свой
  `try/except`: сбой попадает в `report.stage_errors`, но не превращает
  готовый перевод в пустой `status="failed"` результат без единого
  файла — именно так раньше выглядел краш ревью/бэкт-перевода/экспорта
  из-за подвисшего провайдера.

## ✅ Реализовано (v0.11)

Устойчивость к нехватке памяти, спец-OCR-модели и выбор парсера в UI:

- **Memory guard против OOM** — реальный кейс: MinerU почти дотянул
  парсинг (держит гигабайты в дочернем процессе), а Ollama уже начала
  грузить модель перевода — и всё падает по памяти (у пользователя
  съело 36 ГБ). Теперь между парсингом и переводом
  `pipeline._memory_guard` через `resources.wait_for_memory` дожидается,
  пока тяжёлый парсер отдаст RAM, и только потом стартует модель. Порог —
  `PDFTRANSL_MIN_FREE_MEMORY_MB` (≈ размер модели). Память логируется до
  и после парсинга; мало свободной — 🧠-предупреждение в отчёте и UI.
- **Детект залипаний LLM/парсеров** — `resources.Watchdog` следит за
  прогрессом перевода: не завершился ни один сегмент дольше
  `PDFTRANSL_STALL_WARNING_SECONDS` — в лог уходит предупреждение о
  зависшей/неотвечающей модели, а не тихое висение.
- **Мониторинг памяти без зависимостей** — `resources.memory_stats()`
  читает RAM через psutil, а без него — `/proc/meminfo` (Linux) или
  `vm_stat`/`sysctl` (macOS); ничего не доступно — молча выключается.
- **Спец-OCR-модели (DeepSeek-OCR, GOT-OCR, olmOCR, Nanonets-OCR)** —
  распознаются автоматически и получают короткий grounding-промпт вместо
  тяжёлого системного (так они точнее). OCR-модель полностью независима
  от модели перевода: пресет провайдера `deepseek_ocr` (vLLM :8000) для
  парсинга + любая LLM (`PDFTRANSL_MODEL`) для перевода. Свой промпт —
  `PDFTRANSL_OCR_PROMPT`.
- **Выбор парсера прямо в веб-форме** — выпадающий список «Парсер PDF»
  (авто / MinerU / OCR / Nougat / marker / Docling / GROBID / PyMuPDF) и
  отдельное поле «OCR-модель»: не нужно лезть в настройки, чтобы разово
  прогнать документ через OCR или конкретный бэкенд.

## ✅ Реализовано (v0.10)

Возобновление, новые парсеры, очереди и безопасность:

- **Чекпойнты/возобновление** — упавшая задача (падение провайдера,
  крупный файл) при повторном запуске резюмирует уже переведённые
  сегменты из append-only чекпойнта рядом с результатом; переводятся
  только недоделанные. Файл очищается после полного успеха. Заодно
  последовательный режим (`max_workers=1`) стал устойчив к ошибке
  сегмента — как параллельный (`PDFTRANSL_RESUME`).
- **Nougat и GROBID** как бэкенды парсинга (auto-порядок: MinerU →
  Nougat → marker → Docling → GROBID → PyMuPDF). GROBID через сервер
  (`GROBID_URL`), TEI→Markdown.
- **Авто-экспорт датасета для дообучения** — каждые N новых сегментов
  TM выгружается JSONL (`tm_autoexport_every`), sidecar-маркер не даёт
  срабатывать повторно на том же пороге.
- **Очереди Celery по типам нагрузки** — задачи уходят в очередь
  `PDFTRANSL_JOB_QUEUE`, отдельный GPU-воркер можно повесить на тяжёлую
  очередь для MinerU.
- **Усиление загрузок** — валидация PDF по сигнатуре `%PDF-` (не только
  расширению) и опциональный антивирус-хук (`PDFTRANSL_AV_SCAN_CMD`,
  напр. clamscan) — non-zero отклоняет файл.

## Куда двигаться дальше

### Парсинг
- Ансамбль двух парсеров со сверкой формул (расхождения — в отчёт).
- Повторное распознавание битых формул VLM-ом по кропу страницы
  (bbox из `layout.json` MinerU) — сейчас чиним текстово, через LLM.
- Порядок чтения двухколоночной вёрстки и сноски из `middle.json`.

### Перевод и качество
- CometKiwi/BLEURT как альтернатива LLM-судье (без лишних LLM-вызовов,
  но с тяжёлой ML-зависимостью — потому отложено).

### RAG и обучение
- Векторный индекс (sqlite-vec/pgvector/Qdrant) при >50k сегментов —
  numpy-путь уже есть, интерфейс хранилища менять не придётся.

### Выходные форматы
- Наложение перевода на исходную вёрстку по bbox (как BabelDOC) —
  пиксельная реплика оригинального PDF. Самая желанная и самая сложная
  фича этого списка.

### Инфраструктура
- Мультитенантность: пользователи, квоты, учёт стоимости токенов.

### Безопасность
- Полноценные пользователи/сессии вместо общего Bearer-токена.
- Изоляция парсера в контейнере/cgroup (антивирус-скан загрузок уже
  есть через хук).

# Архитектура pdftransl

## Общая схема

```
                     ┌────────────────────────────────────────────┐
                     │              TranslationService            │  <- Django / FastAPI / CLI
                     │   submit() / process() / status() / TM API │
                     └──────────────────┬─────────────────────────┘
                                        │
                     ┌──────────────────▼─────────────────────────┐
                     │            TranslationPipeline             │
                     └────┬─────────┬─────────┬─────────┬─────────┘
                          │         │         │         │
        ┌─────────────────▼──┐ ┌────▼────┐ ┌──▼──────┐ ┌▼──────────────┐
        │ parsing            │ │ transla-│ │ quality │ │ rag           │
        │ MinerU local/API   │ │ tion    │ │ valida- │ │ TM + glossary │
        │ PyMuPDF fallback   │ │ LLM/VLM │ │ tors +  │ │ + embeddings  │
        │ splitter (blocks)  │ │ masking │ │ reviewer│ │ + retriever   │
        └────────────────────┘ └─────────┘ └─────────┘ └───────────────┘
                          │         │         │         │
                     ┌────▼─────────▼─────────▼─────────▼─────────┐
                     │        storage (SQLite): jobs, TM,         │
                     │        glossary, отчёты, артефакты         │
                     └────────────────────────────────────────────┘
```

## Стадии пайплайна

1. **parse** — `parsing/` превращает PDF в Markdown с LaTeX-формулами.
   Бэкенды взаимозаменяемы (`ParserBackend`): локальный MinerU CLI,
   облачный MinerU API, фолбэк PyMuPDF. Картинки/графики экспортируются
   в файлы (`Asset`), ссылки в Markdown сохраняются.
2. **split** — `parsing/splitter.py` разбивает Markdown на типизированные
   блоки (заголовок, абзац, таблица, формула, код, картинка, HTML).
   Блоки формул/кода/картинок помечаются непереводимыми и проходят
   через пайплайн без изменений.
3. **mask** — `masking.py` заменяет в переводимых блоках все
   чувствительные фрагменты (инлайн-формулы, `\cite`, ссылки, URL,
   библиографические номера, HTML-теги) на плейсхолдеры `⟦PHn⟧`.
   После перевода плейсхолдеры восстанавливаются, их целостность —
   первый и самый жёсткий валидатор.
4. **segment** — переводимые блоки группируются в сегменты до
   `chunk_char_budget` символов (блок никогда не режется пополам).
5. **RAG** — `rag/`: для каждого сегмента ищутся
   - точное совпадение в translation memory (перевод переиспользуется
     без обращения к LLM);
   - похожие сегменты (косинусная близость эмбеддингов) — few-shot
     примеры в промпт;
   - термины глоссария, найденные в тексте, — принудительная
     терминология в промпте.
6. **translate** — `translation/translator.py` вызывает LLM через единый
   интерфейс `BaseLLMClient` (`llm/`). После каждого ответа: анмаскинг →
   детерминированные валидаторы → при ошибках цикл исправлений
   (модели отправляется список проблем, до `max_repair_attempts` раз).
7. **review** — `quality/reviewer.py`: LLM-ревьюер перепроверяет
   проблемные сегменты и возвращает вердикт/исправление (JSON).
   Исправление принимается только если не ломает плейсхолдеры.
8. **assemble** — сегменты собираются обратно в Markdown в исходном
   порядке; непереведённые сегменты остаются в оригинале (graceful
   degradation), а их список попадает в `report.json`.
9. **figures (опционально)** — `translation/figures.py`: VLM описывает
   каждый экспортированный рисунок на целевом языке (`figures.json`).
10. **learn** — успешные пары «оригинал → перевод» сохраняются в TM;
    правки человека (`origin=human`) вытесняют автоматические записи
    и имеют приоритет при поиске.

## Ключевые интерфейсы (точки расширения)

| Интерфейс | Файл | Реализации | Как расширить |
|---|---|---|---|
| `ParserBackend` | `parsing/base.py` | MinerU local/API, PyMuPDF | Nougat, marker, GROBID, Docling |
| `BaseLLMClient` | `llm/base.py` | OpenAI-compat, Anthropic, Fake | любой API: реализовать `chat()` |
| `BaseEmbedder` | `rag/embeddings.py` | hashing, sentence-transformers, API | OpenAI/Ollama embeddings и т.д. |
| хранилище TM | `rag/store.py` | SQLite + косинус в Python | pgvector, Qdrant, sqlite-vec |

Провайдеры задаются пресетами в `config.PROVIDER_PRESETS`; любой
OpenAI-совместимый сервер подключается без кода: `--base-url` /
`PDFTRANSL_BASE_URL`.

## Самоконтроль: три уровня

1. **Целостность плейсхолдеров** (ошибка ⇒ ретрай): формулы, ссылки и
   код не могут быть потеряны или придуманы.
2. **Детерминированные валидаторы** (`quality/validators.py`):
   пустой перевод, соотношение длин (обрезание/зацикливание), доля слов
   исходного языка (недопереведённость), количество заголовков, форма
   таблиц, баланс `$$`/`\begin…\end`.
3. **LLM-ревью** (`quality/reviewer.py`): смысловая проверка проблемных
   сегментов вторым проходом модели.

Результат — `report.json`: статистика, список проблемных сегментов с
их issue-кодами, число попыток. Сегменты со статусом «ошибка» остаются
в тексте на исходном языке и явно перечислены в отчёте — ничего не
теряется молча.

## Данные и «обучение»

Всё персистентное состояние — один файл SQLite (`PDFTRANSL_DB`):

- `jobs` — статусы/прогресс/отчёты задач (для веб-поллинга);
- `tm_segments` — память переводов с эмбеддингами (RAG + повторное
  использование);
- `glossary` — терминология.

`pdftransl tm export dataset.jsonl` выгружает память переводов как
датасет — готовую заготовку для LoRA-дообучения локальной модели.

## Интеграция в бэкенд

`TranslationService` разделяет постановку задачи и выполнение:

- `submit()` — быстрый, вызывается из HTTP-запроса;
- `process()` — тяжёлый, вызывается из Celery-воркера / потока /
  management command;
- `status()` — читает прогресс из SQLite (стадия + проценты).

Готовый пример Django-приложения: `integrations/django_example/`
(модель `TranslationJob`, Celery-таска, upload/status/download API,
эндпоинт правок для TM).

# pdftransl

**Переводчик научных PDF, который не ломает формулы.**

Загружаете статью — получаете перевод в Markdown, HTML, DOCX, PDF или
LaTeX, где формулы, таблицы, картинки и структура остались на своих
местах. Работает и с облачными моделями (OpenRouter, OpenAI, Anthropic,
DeepSeek), и полностью локально на вашей машине через Ollama — без
единого API-ключа и без отправки текстов наружу.

Внутри — не «прогнать через ChatGPT», а честный пайплайн: парсинг с
распознаванием формул в LaTeX, защита всего хрупкого плейсхолдерами,
параллельный перевод с самопроверкой и циклом исправлений, LLM-ревью,
память переводов, которая учится на ваших правках, и сборка результата
обратно в приличный документ.

## Самый быстрый способ попробовать

```bash
git clone <репозиторий> && cd pdftransL
./quickstart.sh
```

Скрипт (писался под macOS, работает и на Linux) сам создаст окружение,
поставит зависимости, соберёт веб-интерфейс, настроит `.env` под
локальную Ollama (включая защиту от OOM под размер модели) и подскажет,
какую модель скачать под вашу память. На Windows — то же самое одной
командой в PowerShell:

```powershell
.\quickstart.ps1        # если политика запрещает:
powershell -ExecutionPolicy Bypass -File .\quickstart.ps1
```

Дальше:

```bash
source .venv/bin/activate
pdftransl translate статья.pdf --formats html,docx,pdf
```

Хотите максимальное качество распознавания формул — добавьте MinerU:
`./quickstart.sh --with-mineru` (тяжёлый, зато лучший). Гайд по Ollama
с выбором модели под ваше железо: [docs/OLLAMA.md](docs/OLLAMA.md).

## Как это работает

```
PDF ─→ парсинг (MinerU / Nougat / marker / Docling / GROBID / PyMuPDF;
       сканы и битый текст → VLM/спец-OCR; кэш по хэшу, фолбэк-цепочка)
    ─→ memory guard: ждём освобождения RAM парсером перед загрузкой LLM
    ─→ блоки: заголовки, абзацы, таблицы, формулы, картинки
    ─→ References определяется и не переводится
    ─→ саммари статьи + авто-глоссарий терминов (один LLM-проход)
    ─→ формулы/код/ссылки прячутся за плейсхолдеры ⟦PH42⟧
    ─→ память переводов: точные совпадения — бесплатно, похожие — примеры
    ─→ параллельный перевод (fallback-цепочка провайдеров, rate-limit,
       watchdog на зависшие LLM/парсеры)
    ─→ валидаторы → цикл исправлений → LLM-ревью → LLM-судья (опция)
    ─→ сборка + LLM-починка битых формул + проверка LaTeX
    ─→ экспорт: MD / HTML(KaTeX) / DOCX / PDF / LaTeX / двуязычный MD
    ─→ QA-отчёт + пополнение памяти переводов
```

Главная идея — **плейсхолдеры**: всё, что модель может испортить
(формулы, `\cite{}`, код, ссылки, номера цитирований), заменяется на
токены `⟦PH42⟧` до перевода и побайтово возвращается после. Потерянный
токен — жёсткая ошибка: модель получает список проблем и переводит
заново. Что не удалось починить — остаётся в оригинале и честно
перечисляется в `report.json`, ничего не теряется молча.

**Память переводов** делает систему обучаемой: каждый удачный сегмент
сохраняется, повторные документы переводятся мгновенно и бесплатно,
похожие сегменты подаются модели как примеры, а ваши правки в вычитке
имеют приоритет над автоматикой. Короткие правки-термины автоматически
пополняют глоссарий. Накопили базу — можно выгрузить датасет и дообучить
локальную модель ([docs/FINETUNING.md](docs/FINETUNING.md)).

## Веб-интерфейс

```bash
cd backend && python manage.py runserver   # → http://localhost:8000
```

Три вкладки:

- **Перевод** — загрузка с опциями (в т.ч. выбор парсера PDF —
  MinerU/OCR/… — и отдельной OCR-модели прямо в форме), живой прогресс по
  стадиям (SSE), кнопка **«Пауза»** на запущенной задаче (останавливается
  после текущего сегмента, уже переведённая часть сразу доступна для
  скачивания) и **«Продолжить перевод»** на паузе — без повторного
  перевода готового; скачивание во всех форматах, удаление задач и
  вычитка: оригинал и перевод бок о бок, клик по сегменту — правка —
  «Пересобрать файлы» — и DOCX/PDF обновились уже с вашими правками.
- **Настройки** — серверные умолчания «налету»: провайдер, модели,
  парсер, число воркеров, rpm-лимит, форматы, поведение пайплайна и
  уровень логов. Сохраняются в БД и действуют на все новые задачи сразу,
  без перезапуска; опции в форме перевода всё равно приоритетнее.
- **Глоссарий** — просмотр, добавление и удаление терминов.

В дев-режиме задачи крутятся в фоновых потоках — брокер не нужен. Для
продакшена: `USE_CELERY=1`, redis, `celery -A config worker`. Открываете
API наружу — задайте `PDFTRANSL_API_TOKEN` (Bearer-токен) и помните про
лимит загрузок `PDFTRANSL_UPLOADS_PER_HOUR`.

Хотите видеть всё, что делает сервер, — `PDFTRANSL_LOG_LEVEL=DEBUG`:
каждый LLM-вызов с размерами и временем, решения парсера и экспорта,
ход каждого сегмента. `PDFTRANSL_LOG_FILE=data/pdftransl.log` дублирует
логи в файл с ротацией. Уровень можно переключить и из вкладки
«Настройки» — тоже без перезапуска.

## Телеграм-бот

```bash
TELEGRAM_BOT_TOKEN=... python -m bot
```

Прислали PDF — бот показывает стадии перевода и возвращает файлы.
`/settings` — язык, форматы, провайдер, двуязычный режим на
инлайн-кнопках; настройки запоминаются per-chat.

## CLI на каждый день

```bash
pdftransl translate статья.pdf                          # перевод с дефолтами
pdftransl translate статья.pdf --formats docx,pdf,latex # нужные форматы
pdftransl translate статья.pdf --bilingual              # + документ для вычитки
pdftransl translate статья.pdf --provider ollama --fallback openrouter
pdftransl translate статья.pdf --rpm 15                 # щадим бесплатный тариф
pdftransl translate статья.pdf --score --render-check   # параноидальный режим
pdftransl glossary add "attention head" "головка внимания"
pdftransl tm stats && pdftransl tm export dataset.jsonl
pdftransl engines                                       # что умеет экспорт здесь
```

Все опции доступны и из Python: `TranslationService` /
`TranslationPipeline` (`submit → process → status` для интеграции в свой
бэкенд — см. `pdftransl/service.py`, там докстринг с примером).

## Частые вопросы

**Формулы в DOCX/PDF — читаемые?** Да, во всех форматах. В HTML и PDF
формулы рендерит KaTeX, вшитый в файл (шрифты — data-URI), поэтому всё
работает офлайн и в печати через Chromium, без интернета — HTML
получается полностью автономным. В DOCX без pandoc формулы
вставляются картинками (matplotlib), так что видна настоящая формула, а
не сырой `$...$`. С pandoc — нативные формулы Word (OMML) и полноценная
LaTeX-вёрстка PDF (путь максимального качества, `brew install pandoc`).
`quickstart.sh` сам скачивает Chromium для PDF; вручную —
`python -m playwright install chromium`. `pdftransl engines` покажет,
что реально доступно (движок с несуществующим браузером больше не
заявляется), а если формат запросили без движка — причина будет в
QA-отчёте и в интерфейсе, а не молча пропавший файл.

**Насколько это бесплатно?** Полностью локальный вариант (Ollama +
PyMuPDF/MinerU) не стоит ничего. Облачные модели — по их тарифам, причём
память переводов заметно экономит: повторные и похожие куски не
переводятся заново.

**А если PDF — скан (картинки без текстового слоя)?** pdftransl это
распознаёт: детектор находит скан и, если доступна vision-модель,
автоматически прогоняет страницы через VLM-OCR — распознаёт и текст, и
формулы (в LaTeX). Если ваша **основная** модель мультимодальна
(gemma3, llava, `*-vl`, llama3.2-vision, minicpm-v…), OCR включается сам
— отдельно ничего указывать не нужно. Иначе задайте vision-модель
(`--provider ollama --vision-model qwen2.5-vl`) или выберите
`--backend vlm_ocr`. Нет vision-модели — задача завершится с
предупреждением в отчёте (а не молча пустым переводом).

**На выходе «кракозябры» / бессмысленные символы.** Почти всегда это
битый текстовый слой PDF: русские (и другие не-латинские) шрифты без
корректной ToUnicode-карты — типичная беда PDF с Cyberleninka и
подобных. Страница выглядит нормально, но извлекается мусор (символы
Private Use Area, «·····», replacement-символы), и LLM «переводит» его
в такие же кракозябры. pdftransl теперь это ловит: детектор битого
текста помечает документ, ставит статус «partial» и предлагает включить
OCR (vision-модель / MinerU / `--backend vlm_ocr`) — при доступной
vision-модели переключается на OCR автоматически. Ещё проверьте
направление перевода: если исходник русский, а язык оригинала стоит
`en`, система тоже предупредит.

**MinerU падает/зависает на большом файле.** На Mac без CUDA MinerU
считает OCR на CPU и упирается в свой таймаут на крупных сканах. Это
больше не роняет задачу: парсер откатывается на следующий доступный
бэкенд (при мультимодальной модели — на VLM-OCR), в отчёте видно, что
был фолбэк. Свой лимит — `PDFTRANSL_PARSER_TIMEOUT` (сек). Можно сразу
выбрать более лёгкий парсер: `--backend marker` или `--backend vlm_ocr`.

**Какой парсер выбрать?** MinerU — лучший для формул, но тяжёлый.
Nougat — сквозной OCR формул (нужен GPU). marker — быстрее и легче,
тоже умеет LaTeX. Docling — хорош на таблицах. GROBID — точная
структура и библиография (через сервер `GROBID_URL`). vlm_ocr — для
сканов (распознаёт формулы vision-моделью). PyMuPDF — мгновенный, но
формулы и сканы не распознаёт (текстовые статьи — ок). `--backend auto`
сам возьмёт лучший из установленных, а при сбое одного откатится на
следующий.

**Крупный файл упал на середине.** Ничего страшного — просто запустите
задачу заново: готовые сегменты возьмутся из чекпойнта, переведутся
только оставшиеся (`PDFTRANSL_RESUME`, включено по умолчанию). Чекпойнт
удаляется после полного успеха.

**MinerU почти отработал — и всё упало по памяти (съело 36 ГБ).**
Классический OOM: парсер ещё держит гигабайты, а Ollama/vLLM уже начинает
грузить модель перевода — вдвоём не влезают. Включите memory guard:
`PDFTRANSL_MEMORY_GUARD=true` (по умолчанию) логирует память до и после
парсинга, а `PDFTRANSL_MIN_FREE_MEMORY_MB` заставляет пайплайн **дождаться**
освобождения RAM (после того как парсер отдаст память) перед загрузкой
модели. Поставьте значение примерно в размер модели, напр. `20000` под
модель ~20 ГБ. Если свободной памяти мало, в отчёте и в веб-интерфейсе
появится 🧠-предупреждение. Заодно watchdog (`PDFTRANSL_STALL_WARNING_SECONDS`)
предупредит, если LLM или парсер «залип» и долго не отвечает.

**Хочу распознавать спец-OCR-моделью (DeepSeek-OCR), а переводить другой
LLM.** Так и задумано: OCR-модель выбирается независимо от модели перевода.
Поднимите OCR-эндпоинт (`vllm serve deepseek-ai/DeepSeek-OCR`) и укажите
`PDFTRANSL_PARSER=vlm_ocr`, `PDFTRANSL_VISION_PROVIDER=deepseek_ocr`, а
перевод оставьте на своей LLM (`PDFTRANSL_PROVIDER=ollama`,
`PDFTRANSL_MODEL=qwen2.5:14b`). Спец-OCR-модели (DeepSeek-OCR, GOT-OCR,
olmOCR, Nanonets-OCR) распознаются автоматически и получают короткий
grounding-промпт вместо тяжёлого системного — так они работают точнее. В
веб-форме то же самое: выпадающий список «Парсер PDF» → «OCR» + поле
«OCR-модель».

**Парсинг прошёл нормально, а на выходе — пустой документ (LLM
подтупливала).** Была реальная причина: если модель «зависает» и
отвечает на сегмент пустой строкой (не ошибка, просто ничего), это
раньше протекало в итоговый файл как пустой абзац. Теперь пустой ответ
считается неудачей и откатывается на исходный текст — сегмент, а не
весь документ. Плюс сам документ теперь пишется на диск сразу после
перевода, до необязательных стадий (ревью, оценка качества, обратный
перевод, экспорт) — если одна из них зависнет или упадёт, это попадает
в `report.stage_errors`, но уже переведённый текст никуда не девается.

**Можно поставить задачу на паузу и продолжить позже?** Да — кнопка
«Пауза» в веб-интерфейсе (или `POST /api/jobs/<id>/pause/`)
останавливает перевод после текущего сегмента (в параллельном режиме —
после уже начатых, ещё не запущенные отменяются) и сразу отдаёт
переведённую на этот момент часть документа. «Продолжить перевод»
(`POST /api/jobs/<id>/resume/`) не начинает заново — чекпойнт
(`PDFTRANSL_RESUME`) пропускает уже готовые сегменты и переводит только
оставшиеся. Пауза работает во время перевода; сам парсинг (MinerU и
т.п.) уже ограничен своим таймаутом и не прерывается на середине.

**Прогресс-бар не показывает толком, что происходит.** Теперь показывает:
у каждой задачи свой план стадий, посчитанный из её реальных настроек
(если ревью и обратный перевод выключены — они не занимают место на
баре), и веб-интерфейс рисует его как степпер — пройденные стадии
галочкой, текущая с процентом, остальные ждут. Список задач и прогресс
конкретной задачи обновляются пуш-событиями (SSE), а не постоянными
запросами к серверу по таймеру. Рядом с процентом теперь ещё и
приблизительное время до готовности («осталось ~3 мин») — линейная
экстраполяция «сколько ушло на текущий % — столько же уйдёт на
остаток», как у pip/tqdm; появляется после первых ~3% прогресса и
самокорректируется по ходу выполнения — это оценка, а не гарантия.

**Крупный документ грузит железо (память/CPU) во время перевода.**
Перевод идёт партиями (`PDFTRANSL_TRANSLATE_BATCH_SIZE`, 40 сегментов по
умолчанию) — вместо одного пула потоков на весь документ каждая партия
получает свой, который закрывается перед следующей, а между партиями
документ переписывается на диск и (при включённом `PDFTRANSL_MEMORY_GUARD`)
повторно проверяется свободная память — на случай, если она просела уже
во время самого перевода, а не только перед его стартом.

**Перевод так себе, что подкрутить?** (1) модель побольше; (2) правьте
сегменты в вычитке — правки учат систему; (3) наполните глоссарий своей
терминологией; (4) включите `--score`, чтобы найти слабые места;
(5) накопили TM — дообучите модель ([docs/FINETUNING.md](docs/FINETUNING.md)).
